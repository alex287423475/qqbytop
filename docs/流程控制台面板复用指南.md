# 流程控制台面板复用指南

本文档记录 SEO 文章生产线面板的可复用架构。以后给其他本地运行流程做同款控制台时，不需要重写整套前端，只需要复用 `WorkflowDashboard`，再补齐对应 API、状态文件和本地脚本。

## 1. 通用架构

```text
Next.js Admin Page
  -> WorkflowDashboard 通用面板
  -> dev-only API Routes
  -> 本地脚本执行器
  -> runtime 状态文件
  -> 面板轮询状态、阶段数量、任务卡片、运行日志
```

适合复用的场景：

```text
SEO 文章生产线
竞品采集流程
网页抓取流程
批量翻译预处理流程
文件清洗流程
飞书同步流程
图片批处理流程
```

## 2. 前端复用入口

通用组件：

```text
components/workflow/WorkflowDashboard.tsx
```

SEO 面板只是一个配置示例：

```text
components/pipeline/PipelineDashboard.tsx
```

新增一个流程时，推荐新建：

```text
components/{flow-name}/{FlowName}Dashboard.tsx
app/[locale]/admin/{flow-name}/page.tsx
app/api/{flow-name}/status/route.ts
app/api/{flow-name}/run/route.ts
local-{flow-name}/
```

## 4. 状态 API 协议

`GET /api/{flow-name}/status` 应返回：

```json
{
  "updatedAt": "2026-04-24T01:00:00.000Z",
  "isRunning": false,
  "currentStep": null,
  "lock": null,
  "items": {
    "task-id": {
      "id": "task-id",
      "name": "任务名称",
      "stage": "approved",
      "errors": []
    }
  },
  "counts": {
    "keywords": 3,
    "crawled": 2,
    "cleaned": 2,
    "approved": 1,
    "stored": 0
  },
  "log": [
    {
      "time": "2026-04-24T01:00:00.000Z",
      "step": "crawl",
      "id": "task-id",
      "message": "开始采集：任务名称"
    }
  ]
}
```

为了兼容 SEO 生产线旧字段，`WorkflowDashboard` 也支持 `articles` 字段；新流程统一使用 `items`。

## 5. 运行 API 协议

`POST /api/{flow-name}/run` 请求体：

```json
{
  "step": "crawl",
  "id": "task-id",
  "provider": "mock"
}
```

建议规则：

```text
生产环境返回 403
未知 step 返回 400
已有任务运行时返回 409
执行成功返回 pid、step、success
```

## 6. 输入文件管理扩展

如果流程需要在面板里管理输入文件，比如关键词 CSV、待处理 URL 列表、文件清单，可以启用 `keywordManager`。这个名字来自 SEO 生产线，但本质上是“输入表管理器”。

```tsx
<WorkflowDashboard
  {...props}
  keywordManager={{
    enabled: true,
    apiBase: "/api/pipeline/keywords",
  }}
/>
```

启用后，面板会显示输入文件表格，并支持：

```text
查看当前 CSV 行
新增输入项
删除输入项
预览输入项对应的草稿、已校验文件、已审核文件或已发布文件
```

需要补齐这些 API：

```text
GET    /api/{flow-name}/keywords
POST   /api/{flow-name}/keywords
DELETE /api/{flow-name}/keywords
GET    /api/{flow-name}/keywords/preview?slug=...
```

SEO 生产线当前实现：

```text
lib/pipeline-keywords.ts
app/api/pipeline/keywords/route.ts
app/api/pipeline/keywords/preview/route.ts
```

如果其他流程的输入字段不是 `keyword,slug,locale,category,intent,priority`，优先复制这三个文件后改成对应字段，不要把 SEO 字段硬塞到其他业务里。

## 7. AI 配置扩展

SEO 生产线已经内置本地 AI 配置面板，配置保存到：

```text
local-brain/.env
```

该文件已加入 `.gitignore`，不会提交 API Key。

当前配置 API：

```text
GET  /api/pipeline/ai-config
POST /api/pipeline/ai-config
```

面板字段：

```text
Provider：mock / openai / gemini / claude / deepseek
Base URL：兼容 OpenAI Chat Completions 的网关地址，留空使用官方默认地址
Model：模型名称
API Key：留空保存时保留原密钥，输入空字符串可清空
```

运行流程时，`/api/pipeline/run` 会把这些变量注入本地脚本子进程：

```text
AI_PROVIDER
LLM_BASE_URL
LLM_MODEL
LLM_API_KEY
```

同时兼容 provider 专用变量：

```text
OPENAI_MODEL / GEMINI_MODEL / CLAUDE_MODEL / DEEPSEEK_MODEL
OPENAI_API_KEY / GEMINI_API_KEY / CLAUDE_API_KEY / DEEPSEEK_API_KEY
```


## 9. 复用检查清单

新增流程时按这个顺序做：

```text
1. 定义阶段 stages
2. 定义动作 actions
3. 定义 runtime 状态 JSON
4. 写 status API
5. 写 run API
6. 如果需要输入文件面板，写 keywords/input API
7. 写本地脚本和 status-updater
8. 新建 admin 页面
9. 跑 typecheck 和 build
10. 本地 dev 访问控制台验收
```

第一版优先使用轮询，等流程稳定后再考虑 SSE。
